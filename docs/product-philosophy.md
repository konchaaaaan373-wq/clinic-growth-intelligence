# Neco Clinic Report — プロダクト哲学 / Product Philosophy

このドキュメントは、Neco Clinic Report が「何であって、何でないか」を明確にするための指針です。
機能追加・コピー作成・スコアリング変更を行う際は、必ずこの原則に立ち返ってください。

---

## 1. これは何であって、何でないか

**Neco Clinic Report は、汎用SEOチェッカーではありません。また、URLだけでマーケティング効果を測定するツールでもありません。**

- ❌ 汎用SEOスコアツール（技術的SEO指標を並べるだけのもの）
- ❌ URLを入れるだけで「CPA」「広告効果」「売上貢献」が分かると謳うツール
- ✅ **医療機関の「外部から見える集患力」の準備度診断**であり、HP・SEO・MEO・SNS・医療広告リスク・MMM準備度の各領域における**摩擦（friction）**を特定し、改善の優先度を示すもの

外部から観測できるのはあくまで「構造」と「導線」です。実際に初診が生まれたかどうかは、外部情報だけでは分かりません。
本プロダクトは、その事実に誠実であり続けます。

---

## 2. プロダクト名と役割定義（ブランド体系）

| 名称 | 役割 |
| --- | --- |
| **Neco Clinic Report** | プロダクト全体の名称 |
| **Clinic Report Free** | URLベースの無料診断。外部から見える集患導線・SEO・MEO・SNS接続・医療広告リスク・MMM準備度の**摩擦を特定する外部準備度診断**。効果測定ではない。 |
| **Clinic Report Analytics** | 実データ連携による有料分析レイヤー。日別初診数・チャネル履歴などを取り込み、施策別の初診寄与を推定する。 |
| **Clinic Report MMM** | Clinic Report Analytics の中核機能。日別初診数を目的変数、各チャネル施策を説明変数として、**どの施策が実際の来院に寄与したか**を時系列データから推定する。 |

- **Clinic Report Free = 準備を助ける（外部から見える摩擦の可視化）**
- **Clinic Report Analytics / MMM = 実データで寄与を推定する（有料レイヤー）**

Free は Analytics/MMM への **オンボーディング**（何のデータを集めれば効果測定に進めるか）として機能します。

---

## 3. Free が主張してはならないこと（禁止事項）

Clinic Report Free は、外部から観測できる情報のみを扱います。したがって、以下を **算出・断定してはなりません**。

- ❌ **真の初診CPA**（顧客獲得単価）
- ❌ **初診数への実寄与**（どの施策が何件の初診を生んだか）
- ❌ **ROI / 広告費対効果**
- ❌ 電話予約・直接来院を含む**実成果の測定**
- ❌ 医療広告ガイドラインへの**適合性の保証・法的判断**
- ❌ Googleビジネスプロフィールの**内部パフォーマンス**（口コミ数・表示回数・インサイト等）の取得

Free が出してよいのは、「外部から見える集患ボトルネック」「改善優先度」「MMMを始めるための準備度」までです。
CPA・寄与・ROIは **Clinic Report Analytics / MMM（実データ連携）の領域**であり、そこでも「推定」であって「断定」ではありません。

---

## 4. 未入力の扱い（Missing-input principle）

**未入力は「弱み」ではなく「評価不能（追加情報が必要）」として扱います。**

- SNS URL（YouTube/Instagram/TikTok/LINE）が未入力の場合 → 「SNSが弱い」と断定しない。
  「**未入力です。未運用の可能性もあります。運用中ならURLを追加すると評価できます**」とする。
- 診療科・所在地が未指定の場合（URLのみ診断）→ SEO・MEO・SNS接続・症状ページ評価は
  「**外部から見える範囲での暫定評価**」であることを明示し、情報追加による再診断を促す。
- 「入力あり／HP内リンクなし」と「未入力」と「取得できず未確認」を **区別**する。
  - 未入力 → 評価不能（追加情報が必要）
  - HP内にリンクが無い → 外部からは相互接続が確認できない（改善余地）
  - URL入力ありだが戻し導線なし → 改善余地

原則: **観測できていないことを「無い」と言わない。**

---

## 5. 取得失敗の扱い（Fetch-failure principle）

**サイトを取得できなかった場合、それは「サイト品質が低い」ことではなく「評価不能」です。**

- 総合スコア・ランクを **出さない**（0点・Dランク扱いにしない → 「評価不能」と表示）。
- HP集患導線・SEO・MEO・医療広告リスクは **未評価**（`not_evaluable`）とする。
- サイト内部を見ていない前提の改善提案（例: 「予約導線を固定」「tel:リンクにする」「Web予約導線の設置を最優先」）は **出さない**。
- 代わりに、取得失敗専用の次アクションを出す:
  1. URLを確認して再診断する
  2. 詳細フォームから診療科・所在地・GoogleマップURLを追加する
  3. サイトがBot/外部アクセスをブロックしていないか確認する
- 医療広告リスクは **「注意表現なし」と言わない**。「**本文を取得できなかったため未評価**」とする。

原則: **取得できないことと、品質が低いことは別物である。**

---

## 6. 医療広告リスクの扱い（スクリーニングであって法的判断ではない）

医療広告リスク診断は、**初期スクリーニング**であり、**違反・違法の判定ではありません**。

- 「違反です」「違法です」「修正必須です」「罰則対象です」などの **断定表現は使わない**。
- 使うのは「**要確認の可能性**」「**確認が望ましい**」「**初期スクリーニング**」といった表現。
- 文脈に応じて **優先確認 / 要確認 / 低リスク・文脈確認** に分類する。
  - 「必ず診察のうえで判断します」等の受診促進・「安全に受けられた」等の患者状態の記述 → 低リスク（減点しない）
  - 「安全です」「絶対安全」「副作用なし」等の保証表現 → 優先確認（減点）
- 検出は単語ベースであり、打消し表現・画像内文言・動的描画テキストは対象外という**限界を明示**する。
- 最終確認は **医療広告ガイドラインや専門家（弁護士・広告審査等）** を前提とする。

原則: **法的判断は行わない。改善の初期スクリーニングにとどめる。**

---

## 7. コピーライティングのガードレール

医療機関向けBtoB SaaSとして、誇張せず、信頼を損なわない表現に統一します。

### 使ってよい表現
- 「URLだけで外部集患力を無料診断」
- 「HP・MEO・SNS・医療広告リスクを横断チェック」
- 「初診数MMMに進むための準備度を評価」
- 「外部から見える改善余地を明確化」
- 「実データ連携で施策別の初診寄与を**推定**」

### 使ってはいけない表現
- ❌ 「URLだけでCPAがわかる」
- ❌ 「AIが初診数を正確に予測」
- ❌ 「広告費を必ず削減」
- ❌ 「集患効果を完全に可視化」
- ❌ 「医療広告ガイドライン完全対応」
- ❌ 「効果が**わかる**」（→「外部から見える範囲で**診断**」「効果測定の**準備度**」と言う）

### トーン
- 医師・院長・事務長が見ても違和感がない、落ち着いたBtoB SaaSのトーン
- 「広告代理店っぽい胡散臭さ」を避ける
- 断定より「推定」「準備度」「改善余地」「ボトルネック」

原則: **「効果がわかる」と言い切らない。外部から見える範囲での診断と、効果測定の準備度を語る。**

## 8. ロードマップ（Free → CSVベースMMM → API連携）

Free の外部診断から、実データによる初診寄与推定へ段階的に進みます。

### フェーズ1: Clinic Report Free（提供中）
- URLベースの外部準備度診断。摩擦の可視化と、MMMに必要な不足データの棚卸し。

### フェーズ2: CSVベースの初診数MMM（Clinic Report Analytics / MMM の第一歩）
- **日別初診数のCSVアップロード**を起点にする（最小の実データ連携）。
- あわせて集める説明変数: 診療日/休診日、Google広告費・クリック/表示、医療コラム公開日、
  YouTube/Instagram/TikTok 投稿日・成果、ポスティング実施日/部数/費用、天気/祝日/季節性。
- これらを時系列で突き合わせ、**施策別の初診寄与とおおよそのCPAを「推定」**する。

### フェーズ3: API連携による自動化
- GA4 / Search Console / Google広告 / YouTube Analytics / Googleビジネスプロフィール などを連携し、
  データ収集を自動化する。
- 施策別の推定寄与・推定CPA・来月の施策提案・予算配分提案へ。

### レポート保存・共有の将来計画
- 現状はクライアント状態・localStorage・PDF保存（JSONは開発用）にとどめる。
- 将来的に `/reports/:id` による共有URL＋サーバー側保存、専用PDF生成を検討する。

原則: **URL診断（準備）→ CSVで実データMMM（推定）→ API連携（自動化）。各段で「断定」ではなく「推定」を貫く。**

---

## まとめ（一行で）

> Neco Clinic Report は、外部から見える集患の摩擦を誠実に可視化し（Free）、
> 実データで初診寄与を推定する（Analytics / MMM）ためのプロダクトである。
> 観測していないことは「無い」と言わず、取得できないことを「品質が低い」と言わず、
> 効果を「わかる」と言い切らない。
